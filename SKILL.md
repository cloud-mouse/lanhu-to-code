# lanhu-to-code

根据蓝湖链接或设计稿图片，自动识别项目技术栈，生成 1:1 像素级还原的前端代码。

## 核心原则

1. **像素级还原** — 所有 CSS 值必须与设计稿完全一致，不允许"近似"或"简化"
2. **技术栈适配** — 自动检测项目技术栈，生成对应框架代码
3. **优先复用** — 优先使用项目已有的组件、工具函数、设计 token
4. **完整输出** — 不省略任何代码，不用 `// ...` 占位
5. **元素精准分类** — 严格区分 DOM 节点（文字/色块/线条）与切图资源（图标/背景图/装饰图），分别用代码和图片实现
6. **所见即所出** — 设计稿中存在的每个可见元素都必须还原，不存在的元素绝对不能添加

## 第一步：收集输入

判断用户提供的输入类型：

| 输入类型 | 识别方式 | 处理方式 |
|---------|---------|---------|
| 蓝湖 URL | 包含 `lanhuapp.com` | 调用蓝湖 MCP 获取设计数据 |
| 设计图图片 | 图片文件路径或截图 | 用视觉分析提取设计信息 |
| 纯文本描述 | 文字描述 UI | 追问用户提供设计稿 |

**缺少输入时必须追问：**
```
请提供设计稿信息（以下任一）：
1. 蓝湖页面链接（如 https://lanhuapp.com/web/#/item/project/...）
2. 设计稿截图/图片文件路径
3. 设计稿图片（直接粘贴）
```

---

## 第二步：技术栈检测

运行 `scripts/detect-tech-stack.sh <项目根目录>`，获取项目技术栈信息。

脚本会返回：
```
TECH_STACK: vue3
UI_FRAMEWORK: element-plus
CSS_PREPROCESSOR: scss
STATE_MANAGEMENT: pinia
ROUTER: vue-router
BUILD_TOOL: vite
PACKAGE_MANAGER: pnpm
```

**如果脚本检测失败**，按以下顺序手动检测：
1. 读 `package.json` 的 `dependencies` 和 `devDependencies`
2. 读 `CLAUDE.md` 或 `README.md` 中的技术栈说明
3. 扫描 `src/` 目录结构推断

**技术栈决定代码风格** → 读 `references/framework-patterns.md` 获取对应框架的代码模板。

---

## 第 2.5 步：意图判断（新建 vs 改造）

在获取设计数据之前，先判断用户意图：

| 依据 | 判断 |
|------|------|
| 用户明确说"修改/改造/更新 xxx 页面" | **改造模式**，记录目标文件路径 |
| 用户指定了现有文件路径 | **改造模式** |
| 项目有 `docs/tech-spec.md`，且设计稿页面名匹配已有路由 | **改造模式**，自动定位文件 |
| 以上均不满足 | **新建模式** |

**自动匹配逻辑（有 tech-spec 时）：**
1. 读取 `docs/tech-spec.md` 中的路由/页面列表
2. 将用户提到的页面名与路由列表比对
3. 找到匹配 → 改造模式，记录文件路径
4. 未找到 → 新建模式

**无法判断时向用户询问：**
```
请确认操作类型：
1. 新建页面（根据设计稿生成新文件）
2. 改造现有页面（请提供现有文件路径，如 src/pages/home/index.vue）
```

**意图判断结果会传递给第九步，决定走"新建页面模式"还是"改造现有页面模式"。**

## 第三步：获取设计数据

### 输入 A：蓝湖 URL

1. **判断 URL 是否指向具体页面**
   - URL 含 `&id=` 或 `&did=` → 直接使用
   - URL 只有 `pid=` → 调用 `lanhu_design` 的 `list` 模式展示设计列表，等用户选择

2. **获取设计数据（使用 compact 模式防止上下文爆炸）**
   调用 `mcp__lanhu__lanhu_design`，参数：
   - `url`: 蓝湖页面 URL
   - `mode`: "analyze"
   - `design_names`: 具体页面名或序号
   - `include`: ["html", "tokens", "layers", "image"]
   - `compact`: true — **关键！** HTML 代码保存到文件不内联到上下文，避免触发自动压缩

3. **获取切图**（如有）
   调用 `mcp__lanhu__lanhu_design`，参数：
   - `mode`: "slices"
   - `design_names`: 同上

4. **保存产物**到 `.claude/lanhu-output/<页面名称>/`

5. **视觉校验（必须执行，在进入第四步之前）**

   拿到设计稿数据后，**先看设计稿完整截图，再看 HTML/Tokens 数据**。这一步用于建立整体视觉认知，避免后续遗漏：

   a. **分析设计稿截图** — 使用 `mcp__zai-mcp-server__analyze_image` 或 `mcp__4_5v_mcp__analyze_image` 分析 MCP 返回的设计稿截图，逐区域扫视，建立视觉认知：
      - 页面整体背景色/渐变是什么？
      - 哪些区域有明显的背景色或渐变背景？
      - 哪些元素有明显的阴影、圆角、边框？
      - 整体色调和视觉层次是怎样的？
      - 各区域的标题/徽章/按钮的对齐方式是什么（居中、左对齐、右对齐）？
      - 各元素的布局方向（水平排列、垂直堆叠、重叠）是否明确？

   b. **用 Design Tokens 补充细节** — 对照截图，逐区域核实 tokens 中的颜色/渐变/阴影值：
      - 页面背景色是否在 tokens 中体现？
      - 每个区域的背景色/渐变背景色是否都有对应的 token？
      - 按钮渐变、卡片渐变、导航栏渐变是否都提取到了？
      - **冲突处理：当 tokens/标注数据与截图视觉不一致时（如 tokens 返回白色但截图明显是橙色），以截图为准，用图像分析工具从截图中提取准确色值。**
      - **缺失兜底：tokens 中缺失的颜色/渐变值（如按钮渐变色不在 tokens 中），必须用图像分析工具从设计稿截图中提取，禁止用其他页面的值或凭经验猜测代替。**

   c. **输出视觉校验清单**：
      ```
      ## 视觉校验清单

      | 区域 | 视觉感知 | Tokens 值 | 冲突处理 | 状态 |
      |------|---------|-----------|---------|------|
      | 页面背景 | 橙色 | #FFFFFF（❌不匹配） | 以截图为准，提取为 #FF770F | ✓ |
      | 头部导航栏 | 深色渐变 | linear-gradient(135deg, #667eea, #764ba2) | 无冲突 | ✓ |
      | 卡片徽章渐变 | 橙黄渐变 | 缺失 | 从截图提取 → linear-gradient(94deg, #FFB843, #FF5C29) | ✓ |
      | 卡片标题对齐 | 居中 | 无方向数据 | 从截图确认 → text-align: center | ✓ |
      | ... | ... | ... | ... | ... |
      ```

   **这条规则的目的是：** 避免"只看代码不看图"导致的背景色遗漏、渐变色错误、布局方向误判。截图的视觉冲击力比代码更直观，先看图再对数据，能快速发现遗漏和冲突。**当数据与截图矛盾时，永远相信截图。**

**上下文管理：** compact 模式下，设计稿 HTML/CSS 代码保存到磁盘文件，不会加载到对话上下文中。后续步骤（4-9）需要 CSS 数值时，从磁盘读取 `.claude/lanhu-output/<页面名称>/` 中已保存的 HTML 文件，而非依赖上下文中的内联数据。

### 输入 B：设计图图片

1. **分析图片**
   对图片进行详细分析，提取：
   - 整体布局结构（flex/grid、行列、间距）
   - 颜色值（背景、文字、边框、渐变）
   - 字体信息（字号、字重、行高）
   - 间距值（margin、padding、gap）
   - 圆角、阴影、边框
   - 图标和图片资源

2. **后续步骤同蓝湖流程，但需降级处理** → 读 `references/image-input-guide.md`
   - 图片无 layer tree / tokens / 坐标数据，所有数值靠视觉分析提取
   - 切图资源无法自动下载，标注缺失由设计人员补充
   - 输出时告知用户图片输入的精度上限低于蓝湖输入

## 第四步：页面整体属性提取（强制检查点 #1）

**在任何元素分析之前，先提取页面级别的全局属性。这是防止页面背景色丢失的关键步骤。**
**数据来源：从第三步保存的 HTML 文件中读取页面根元素的 CSS 属性。**

### 4.1 页面背景色

从设计稿最底层/画布层级提取：

```
## 页面全局属性

| 属性 | 设计稿值 | 说明 |
|------|----------|------|
| 页面背景色 | #F5F5F5 / linear-gradient(...) | 画布最底层填充色 |
| 页面最小高度 | 100vh / auto | 是否撑满全屏 |
| 内容区最大宽度 | 750rpx / 1200px | 内容约束 |
```

**常见遗漏：**
- 页面整体渐变背景（不仅仅是顶部 hero 区域的背景）
- body/html 级别的背景色
- 移动端安全区域背景色

### 4.2 页面尺寸基准

| 项目 | 提取方式 |
|------|---------|
| 设计稿物理宽度 | 蓝湖画布宽度（如 750px） |
| 适配方案 | 375 基准 → rpx，750 基准 → rpx 1:1，PC → px |
| 状态栏处理 | 是否自定义导航栏 |

## 第五步：元素分类与勾选清单（强制检查点 #2）

**这是防止元素遗漏和类型误判的核心步骤。在设计稿的所有 layers/tokens 数据中，逐元素进行分类。**

分类规则、决策树、勾选清单格式、切图处理方式 → 读 `references/element-classification.md`

**数据来源：从保存的 HTML 文件中读取所有元素的 CSS 类和属性，结合 Design Tokens 补充渐变/边框/阴影信息。**

**输出勾选清单后，等待用户确认再继续。**

---

## 第六步：CSS 值对照表（强制检查点 #3）

**在写任何样式代码之前，必须完成此步骤。**
**数据来源：从保存的 HTML 文件中逐一提取每个 CSS 类的属性值，不要凭记忆或截图猜测。**

子元素粒度颜色提取、布局方向确认、数据驱动尺寸计算等详细规则 → 读 `references/css-value-extraction.md`

### 对照表格式

```
## CSS 值对照表

| 选择器 | 属性 | 设计稿值 | 代码值（待填） | 来源 |
|--------|------|----------|---------------|------|
| .header | background | linear-gradient(135deg, #667eea 0%, #764ba2 100%) | — | 蓝湖/图片 |
| .title | font-size | 32px | — | 蓝湖/图片 |
```

**填写"代码值"列时：**
1. 先查项目的 CSS 变量 / 设计 token（如 `var(--color-primary)`、`$primary-color`）
2. 有对应 token → 用 token
3. 无对应 token → 用设计稿原始值，并在输出中标注 `⚠ 未映射 token`
4. **禁止简化**：渐变不能用纯色代替，多阴影不能省略

输出对照表后，**等待用户确认**再继续。

## 第七步：项目上下文收集

按以下优先级读取项目文档：

| 用途 | 优先查找 | 兜底 |
|------|---------|------|
| 组件库 | `docs/components.md` → `docs/开发规范与组件文档.md` | 扫描 `src/components/` |
| 编码规范 | `docs/dev-spec.md` → `docs/` 下含"规范"的 .md | 读现有文件推断风格 |
| 设计 token | `src/styles/variables.scss` → `tailwind.config.*` → `theme.*` | 无则用设计稿原始值 |
| 路由配置 | `src/router/` → `docs/tech-spec.md` | 生成新路由 |

**读取已有的设计 token 文件**（关键步骤）：
- 查找 `variables.scss` / `variables.css` / `theme.ts` / `tailwind.config.*`
- 提取已有的颜色、间距、字号变量
- 在代码中优先使用这些变量

## 第八步：组件映射

**在映射组件之前，必须先完成 layer tree 结构分析：**

### 8.0 Layer Tree 结构还原

从 layer tree 中理解元素的嵌套和兄弟关系，确定 HTML 结构：
**数据来源：从保存的 HTML 文件的 DOM 结构和 CSS 类名中推断层级关系。**

1. **读取 layer tree 的层级关系**
   - 父子关系 → 嵌套的 HTML 容器
   - 兄弟关系 → 同级 HTML 元素

2. **识别视觉重叠的兄弟元素**
   - 如果两个兄弟元素在 Y 轴上有重叠（如图标"坐"在进度条上）
   - 必须用 `position: relative` + `position: absolute` 实现重叠
   - **不能把所有元素都放进 flex column 顺序排列**

3. **输出 HTML 结构草图**
   ```
   milestone-labels (flex row)
     milestone-btn × 3
   progress-section (position: relative)
     progress-track (绝对定位背景条)
     progress-icon × 3 (绝对定位，坐在进度条上)
   milestone-statuses (flex row)
     status-text × 3
   ```

### 8.1 组件查找与映射

对设计稿中的每个 UI 元素：

1. **查找项目已有组件**
   - 搜索 `src/components/` 下的组件名和 props
   - 检查 UI 框架（Element Plus、Ant Design、Vant 等）的组件

2. **映射规则**
   | 情况 | 处理 |
   |------|------|
   | 项目有完全匹配的组件 | 直接用，传正确 props |
   | UI 框架有对应组件 | 用框架组件，按设计稿覆盖样式 |
   | 无匹配 | 新建组件，放入 `src/components/`，标注 `⚠ 新增组件` |

3. **常见错误规避** → 读 `references/pitfalls.md`

## 第九步：生成代码

### 新建页面模式

当判断为新页面时：

1. 确定输出路径（遵循项目路由约定）：
   - Vue: `src/pages/<name>/index.vue` 或 `src/views/<name>/index.vue`
   - React: `src/pages/<name>/index.tsx`
   - uni-app: `src/pages/<name>/index.vue`

2. 按检测到的技术栈生成代码 → 使用 `references/framework-patterns.md` 中的模板

3. **代码生成顺序（严格按此顺序写）：**
   a. **页面容器** — 最外层 view/div，设置页面背景色、最小高度
   b. **区域容器** — 按设计稿从上到下的区域划分
   c. **区域内部元素** — 按勾选清单逐个还原，先 DOM 元素再切图元素
   d. **交互逻辑** — script/setup 中的状态和事件
   e. **路由配置** — 如需要

4. **生成内容：**
   - 完整的 template/JSX
   - 完整的 script（含状态、事件处理、生命周期）
   - 完整的 style（像素级还原所有样式）
   - 组件引入和注册
   - 路由配置（如需要）

### 改造现有页面模式

当用户指定要修改的文件时：

1. 读取现有文件，理解现有逻辑
2. 列出变更范围，等用户确认：
   ```
   变更项：
   - [新增] 顶部搜索区域
   - [修改] 列表项间距从 12px 改为 16px
   - [删除] 底部多余分割线
   - [保留] 所有 API 调用逻辑
   ```
3. 只修改变更项，保留现有业务逻辑

## 第十步：输出与检查

### 输出清单

```
✓ 生成文件：[文件列表]
✓ 复用组件：[组件列表]
⚠ 新增组件：[需 review 的组件列表]
⚠ 未映射 token：[未映射的设计值列表]
⚠ TODO 项：[需确认的逻辑列表]
```

### 自检清单（生成后必须过一遍）

- [ ] **页面背景色**与设计稿一致（检查最外层容器的 background）
- [ ] **元素勾选清单**中所有项均已实现（逐项核对，无遗漏）
- [ ] 所有 CSS 值与设计稿一致（对照表逐一核对）
- [ ] **颜色粒度**：每个子元素的颜色都与设计稿匹配（非区域主色）
- [ ] **布局方向**：元素的上下左右位置关系与设计稿一致
- [ ] **内容区重叠量**：从设计坐标反推的负 margin 值，不是猜的
- [ ] **每张卡片 padding 逐一计算**：不是统一值
- [ ] **元素尺寸从数据提取**：进度条高度、图标大小、缺口大小等来自 layer tree，不是感觉值
- [ ] **相邻元素间距**：从绝对坐标差值计算，不是 margin-top 链式累加
- [ ] **Layer tree 层叠关系**：视觉重叠的兄弟元素用 absolute 定位实现
- [ ] **切图已含背景的不再加背景**：如步骤图标切片已含圆形背景
- [ ] 切图资源已下载到本地并正确引用
- [ ] 无 `// ...` 省略
- [ ] 无自行发明的 API 路径（用 TODO 标记）
- [ ] **无设计稿中不存在的元素**（标题、分隔线、装饰等）
- [ ] 代码符合项目编码规范（缩进、引号、分号等）
- [ ] 组件 props 名称与组件文档一致

### 视觉复核（如有设计稿图片）

将生成的代码结构与设计稿图片逐区域比对：
1. 从上到下扫一遍，对比每个区域的元素构成
2. 重点检查：布局方向、元素遗漏、颜色差异、间距偏差
3. 标注偏差项并告知用户

## 禁止事项

| 禁止 | 原因 |
|------|------|
| 简化渐变为纯色 | 破坏设计还原度 |
| 省略代码用占位符替代 | 输出不可用 |
| 自行发明 API 路径 | 产生 bug |
| 跳过 CSS 对照表直接写代码 | 导致还原度低 |
| 猜测组件 props 名称 | 使用错误 API |
| **添加设计稿中不存在的任何元素** | **包括但不限于：标题行、分隔线、图标、按钮、装饰性 UI。如果不确定某元素是否在设计中存在，标注 `⚠ 未在设计中确认`，不要自行添加** |
| 忽略项目现有组件直接写原生 HTML | 重复造轮子 |
| **用区域主色覆盖子元素颜色** | **导致颜色不精确（如优惠券金额应为红色却被标为金色）** |
| **凭直觉判断元素布局方向** | **导致上下左右位置错误。必须以设计稿坐标数据为准** |
| **跳过页面背景色提取** | **导致页面整体色调丢失** |

# lanhu-to-code

> v2.0.0

根据蓝湖链接或设计稿图片，自动识别项目技术栈，生成 1:1 像素级还原的前端代码。

## 核心原则

1. **像素级还原** — 所有 CSS 值必须与设计稿完全一致，不允许"近似"或"简化"
2. **技术栈适配** — 自动检测项目技术栈，生成对应框架代码
3. **优先复用** — 优先使用项目已有的组件、工具函数、设计 token
4. **完整输出** — 不省略任何代码，不用 `// ...` 占位
5. **所见即所出** — 设计稿中存在的每个可见元素都必须还原，不存在的元素绝对不能添加

---

## 流程概览

```
[1] 数据准备（并行）— 技术栈 + 设计数据 + 切图 + 项目上下文
[2] 代码生成 — 边读值边生成，只对重叠区域做坐标计算，完成后写缓存
[3] 验证 — 编译 + 自检
```

两轮策略：
- **第一轮（快速生成）**：直接从 annotations 读值生成代码，目标快速出结果
- **第二轮（定点修正）**：用户反馈后，只修正问题区域，利用缓存精确计算

---

## 第 1 步：数据准备（并行）

### 1.1 输入判断

| 输入类型 | 识别方式 | 处理方式 |
|---------|---------|---------|
| 蓝湖 URL | 包含 `lanhuapp.com` | 调用蓝湖 MCP 获取设计数据 |
| 设计图图片 | 图片文件路径或截图 | 用视觉分析提取设计信息 |
| 纯文本描述 | 文字描述 UI | 追问用户提供设计稿 |

### 1.2 意图判断（新建 vs 改造）

用户说"修改/改造"或指定文件路径 → **改造模式**。否则 → **新建模式**。

### 1.3 并行任务组

以下任务全部并行执行，一批完成：

```
并行任务 A: 技术栈检测
  → 运行 bash scripts/detect-tech-stack.sh，失败时手动检查 package.json
  → 按需读 references/framework-patterns.md

并行任务 B: 蓝湖数据获取
  → detailDetach URL: 先 list 获取 design_names，再并行调 analyze + slices
  → stage URL: 直接并行调 analyze + slices
  → analyze 参数: include: ["html", "tokens", "layers", "image"], design_names: "<名称>"

并行任务 C: 项目上下文（Agent Explore）
  → 扫描 src/components/、docs/、src/styles/variables.scss、路由配置
  → 意图为改造时，读取目标文件理解现有逻辑
```

### 1.4 下载切图

获取 slices 后，筛选需要下载的切图（排除状态栏电池/WiFi 等系统 UI 小零件），批量下载到项目静态资源目录。

### 1.5 图片输入流程

输入是图片（非蓝湖 URL）时，用视觉分析提取布局、颜色、字体信息 → 后续步骤降级处理（参见 `references/image-input-guide.md`）。

## 第 2 步：代码生成

### 核心策略：边读值边生成，不预先分析

**不设独立的"设计分析"步骤。** 直接从 annotations/tokens 读取 CSS 值，边读边写代码。只对重叠区域做坐标计算。

### 2.1 页面全局属性

从设计稿最底层提取：页面背景色、最小高度、设计稿宽度、适配方案（rpx/px）。直接写进页面容器样式。

### 2.2 按区域从上到下生成

对设计稿的每个区域，先判断布局类型再生成：

**流式布局（90% 的区域）— 直接读值，用 flex+gap+padding：**

| 需要的值 | 从哪读 | 换算 |
|---------|--------|------|
| 容器尺寸 | layer_tree `(702x294)` | 直接是 rpx |
| padding | annotations 容器位置减内容位置 | 逻辑px × 2 |
| gap | 同行相邻元素的位置差 | 逻辑px × 2 |
| 文字颜色 | textLayer 的 color | 直接使用 |
| 字体大小 | textLayer 的 font-size | 逻辑px × 2 |
| 背景色 | shapeLayer 的 fill | 直接使用 |

**重叠布局（少数区域）— 精确坐标计算：**

只有以下场景才做坐标计算：
- **区域之间的重叠量**（如内容卡片与 hero 区域的负 margin）
- **absolute 定位的装饰元素**（如卡片内的装饰图片）
- **兄弟元素有视觉交叉**（如图标"坐"在进度条上）

计算公式：
```
重叠量 = 上方元素 bottom - 下方元素 top（均为绝对坐标 rpx）
间距 = next.top - prev.top - prev.height
padding = (外框宽 - 内容宽) / 2
```

### 2.3 元素分类（边生成边判断）

| 分类 | 判断依据 | 实现方式 |
|------|---------|---------|
| DOM 文字 | 纯文本，有字体/字号/颜色 | HTML + CSS |
| DOM 形状 | 简单几何（矩形/圆/线条），纯色或渐变 | HTML + CSS |
| 切图-图标 | < 100px，有明确语义 | `<image>` + 切图文件 |
| 切图-背景 | 大面积，含丰富色彩/图案 | `<image>` + 背景色兜底 |

### 2.4 渐变色处理

1. 标注中有具体渐变色值 → 直接使用
2. 标注只标 `gradient:linear` 无色值 → 代码中写 `/* TODO: 渐变色需视觉确认 */` 占位，用合理近似值
3. 切图/装饰图已含渐变 → 直接用切图，CSS 不重复写渐变
4. 文字渐变 → `background-clip: text` 实现

### 2.5 生成顺序

1. 页面容器 — 最外层，设置背景色、最小高度
2. 区域容器 — 从上到下
3. 区域内部元素 — 按元素分类逐个还原
4. 交互逻辑 — script 中的状态和事件
5. 路由配置 — 如需要

### 2.6 写入设计数据缓存

代码生成完成后，将关键 CSS 值写入 `.claude/lanhu-cache/<design-name>.md`：

```markdown
# <设计稿名称>
## 页面属性
- 画布: <宽>×<高>rpx, 背景 <色值>, 适配 <rpx/px>

## 区域关键值
- <区域名>: <尺寸>, margin <值>, padding <值>, border-radius <值>
- <重叠关系>: <重叠量>rpx

## 渐变色（来自视觉分析）
- <元素>: linear-gradient(<方向>, <色值们>)
```

此缓存文件在上下文压缩后仍可用，避免重复获取设计数据。

### 2.7 改造现有页面

1. 读取现有文件，理解现有逻辑
2. 列出变更范围，等用户确认
3. 只修改变更项，保留现有业务逻辑

## 第 3 步：验证

### 3.1 编译验证

- uni-app：`npx uni build -p mp-weixin`（用 build 不用 dev，dev 会 hang）
- Vue/React/Vite：`npm run build`（不用 dev server，会 hang）
- 仅验证编译通过无语法错误，不做运行时验证

### 3.2 自检清单

**结构完整性：**
- [ ] 页面背景色与设计稿一致
- [ ] 所有元素均已实现，无遗漏
- [ ] 无省略占位符，无设计稿中不存在的元素
- [ ] 切图资源已下载到本地并正确引用

**CSS 精确度：**
- [ ] 颜色粒度：每个子元素颜色匹配（非区域主色）
- [ ] 重叠区域：重叠量从坐标精确计算，非估算值
- [ ] 流式区域：未做坐标计算，用 flex+gap+padding
- [ ] 渐变色来源已标注

**四项校验：**
- [ ] 可见性：文字颜色不与父容器背景色相同导致不可见
- [ ] 非跨页复制：借用的值已与当前设计数据验证
- [ ] 富文本：含数字/状态词的文本已逐字核对颜色
- [ ] 切图尺寸：含背景切图不加额外背景，不含背景切图加对应容器

### 3.3 定点修正（第二轮）

用户反馈后：
1. 只读取问题区域的设计数据（从缓存文件或重新查 annotations）
2. 只修正问题区域的 CSS 值
3. 执行四项校验
4. 编译验证

## 组件映射

- 项目有完全匹配的组件 → 直接用
- UI 框架有对应组件 → 用框架组件，覆盖样式
- 无匹配 → 新建组件，标注 `⚠ 新增组件`

## 禁止事项

| 禁止 | 原因 |
|------|------|
| 简化渐变为纯色 | 破坏还原度 |
| 省略代码用占位符 | 输出不可用 |
| 自行发明 API 路径 | 产生 bug |
| 猜测组件 props 名称 | 使用错误 API |
| 添加设计稿中不存在的元素 | 还原度失真 |
| 忽略项目现有组件直接写原生 HTML | 重复造轮子 |
| 用区域主色覆盖子元素颜色 | 颜色不精确 |
| 凭直觉判断布局方向 | 位置错误 |
| 跳过页面背景色提取 | 色调丢失 |
| 从其他页面复制 CSS 值时未与当前设计数据验证 | 值可能不同 |
| 文字颜色与父容器背景色相同时未标记告警 | 不可见问题 |
| 对含数字/状态词的文本不核对逐字颜色 | 富文本丢失 |
| **对流式布局区域做逐像素坐标推算** | **浪费时间，flex 即可还原** |
| **调用额外的视觉分析工具做"视觉校验"** | **蓝湖 analyze 已含精确数据，视觉分析返回泛泛描述，ROI 低** |
| **预先完成所有分析再写代码** | **应边读值边生成，减少步骤间数据遗忘** |

---

## 性能规范

1. **最多 3 批工具调用**：
   - 第 1 批：技术栈检测 + 蓝湖 MCP（analyze + slices） + 项目上下文（Agent Explore）
   - 第 2 批：下载切图 + 生成代码 + 写缓存
   - 第 3 批：编译验证
2. **不调用额外视觉分析工具** — 蓝湖 analyze 已包含 tokens + annotations + layer tree + 设计截图，足够还原
3. **流式区域零坐标计算** — 导航栏、卡片内部、网格、底部栏等全部用 flex+gap，禁止计算任何绝对坐标
4. **缓存必写** — 每次生成后必须写缓存文件，确保上下文压缩后不丢失数据
5. **定点修正不全量重写** — 用户反馈后只改问题区域

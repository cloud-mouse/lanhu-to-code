# lanhu-to-code

> v1.1.0

根据蓝湖链接或设计稿图片，自动识别项目技术栈，生成 1:1 像素级还原的前端代码。

## 核心原则

1. **像素级还原** — 所有 CSS 值必须与设计稿完全一致，不允许"近似"或"简化"
2. **技术栈适配** — 自动检测项目技术栈，生成对应框架代码
3. **优先复用** — 优先使用项目已有的组件、工具函数、设计 token
4. **完整输出** — 不省略任何代码，不用 `// ...` 占位
5. **所见即所出** — 设计稿中存在的每个可见元素都必须还原，不存在的元素绝对不能添加

---

## 流程概览

```
[1] 输入判断 + 技术栈检测 + 意图判断（并行）
[2] 获取设计数据 + 下载切图 + 视觉校验
[3] 设计分析（页面属性 + 元素分类 + CSS 值 + 四项校验）
[4] 项目上下文 + 组件映射（与步骤 3 并行）
[5] 生成代码 + 自检
```

---

## 第 1 步：收集输入 + 技术栈检测（并行）

### 输入判断

| 输入类型 | 识别方式 | 处理方式 |
|---------|---------|---------|
| 蓝湖 URL | 包含 `lanhuapp.com` | 调用蓝湖 MCP 获取设计数据 |
| 设计图图片 | 图片文件路径或截图 | 用视觉分析提取设计信息 |
| 纯文本描述 | 文字描述 UI | 追问用户提供设计稿 |

### 技术栈检测

通过 Bash 工具运行 `bash scripts/detect-tech-stack.sh <项目根目录>`（脚本位于 skill 安装目录下）。失败时手动检查 `package.json` → `CLAUDE.md` → `src/` 目录结构。技术栈决定代码风格 → 按需读 `references/framework-patterns.md`。

### 意图判断（新建 vs 改造）

用户说"修改/改造"或指定文件路径 → **改造模式**。否则 → **新建模式**。

---

## 第 2 步：获取设计数据

### 蓝湖 URL 流程

1. **获取设计数据** — 调用 `mcp__lanhu__lanhu_design`，参数 `mode: "analyze"`, `include: ["html", "tokens", "layers", "image"]`
2. **获取切图** — 同步调用 `mode: "slices"`
3. **保存产物**到 `.claude/lanhu-output/<页面名称>/`
4. **下载关键切图**到项目静态资源目录

### 视觉校验（获取数据后立即执行）

先看设计稿截图，再看 tokens/标注数据：

a. **分析截图** — 用图像分析工具逐区域扫视：背景色、渐变、阴影、圆角、布局方向
b. **核对 tokens** — 截图与 tokens 冲突时，以截图为准
c. **关键检查**：
   - 每个区域的背景色/渐变是否都有对应 token？缺失的从截图提取
   - 卡片圆角、按钮圆角在截图中是否可见？tokens 无此值时从截图提取，**禁止从其他页面推断**
   - tokens 中缺失的值，必须用图像分析工具从截图中提取

### 图片输入流程

用视觉分析提取布局、颜色、字体信息 → 后续步骤降级处理（参见 `references/image-input-guide.md`）。

---

## 第 3 步：设计分析

### 3.1 页面全局属性

从设计稿最底层提取页面背景色、最小高度、设计稿宽度、适配方案（rpx/px）。

### 3.2 元素分类

| 分类 | 判断依据 | 实现方式 |
|------|---------|---------|
| DOM 文字 | 纯文本，有字体/字号/颜色 | HTML + CSS |
| DOM 形状 | 简单几何（矩形/圆/线条），纯色或渐变 | HTML + CSS |
| 切图-图标 | < 100px，有明确语义 | `<image>` + 切图文件 |
| 切图-背景 | 大面积，含丰富色彩/图案 | `<image>` + 背景色兜底 |

### 3.3 CSS 值提取

从设计标注中逐一提取。**核心规则：**

- 颜色按**最小可见单元**逐一提取（不取区域主色）
- 间距从**绝对坐标差值**计算（不用 margin-top 链式累加）
- 尺寸从 **layer tree / annotations** 提取（不凭感觉）
- 每张卡片 **padding 逐一计算**
- 优先使用项目的 CSS 变量 / 设计 token，无对应 token 时用设计稿原始值

### 3.4 CSS 冲突速查表（异常时输出）

正常情况下不输出对照表，直接内化到代码中。但遇到以下情况时，**必须输出精简版冲突表**供用户审查：

- tokens 数据与截图视觉不一致（颜色、圆角、间距）
- 从已有页面借用的值与当前设计数据不同
- 四项校验中发现异常（可见性问题、富文本变色、切图尺寸偏差）

无冲突时跳过此表，避免输出噪音。

### 3.5 四项强制校验（防止高频 bug）

**CSS 值提取完成后，必须逐条过以下校验。任何一项失败都需修正后再继续。** 详细规则和示例见 `references/css-value-extraction.md`。

| 校验 | 核心规则 | 常见问题 |
|------|---------|---------|
| **A. 可见性** | 文字颜色 ≈ 父容器背景色 → 标记 ⚠，检查兄弟节点是否有背景 shape | 白字白底：文字实际有渐变 pill 包裹 |
| **B. 跨页复制** | 借用已有页面的具体 CSS 值前，必查当前 tokens/annotations 是否有不同值 | 不同设计稿的圆角/间距可能不同 |
| **C. 富文本** | 含数字/状态词的文本，用图像分析逐字核对颜色，拆分为多个元素 | "3人"应为红色但标注整体灰色 |
| **D. 切图尺寸** | 下载切图后检查是否含背景，决定 CSS 尺寸和背景写法 | artboard 含背景直接展示，shapeLayer 需额外背景容器 |

### 3.6 Layer Tree 结构还原

- 父子关系 → 嵌套的 HTML 容器
- 兄弟关系 + 视觉重叠 → `position: relative` + `position: absolute`
- 不能把所有元素都放进 flex column 顺序排列

## 第 4 步：项目上下文 + 组件映射（与第 3 步并行）

### 读取项目文档

| 用途 | 优先查找 | 兜底 |
|------|---------|------|
| 组件库 | `docs/components.md` | 扫描 `src/components/` |
| 编码规范 | `docs/dev-spec.md` | 读现有文件推断风格 |
| 设计 token | `src/styles/variables.scss` → `tailwind.config.*` | 用设计稿原始值 |
| 路由配置 | `src/router/` → `docs/tech-spec.md` | 生成新路由 |

**注意校验 B：参考已有页面的代码风格（组件结构、命名规范），但具体 CSS 值（圆角、间距、尺寸）必须以当前设计稿数据为准。**

### 组件映射

- 项目有完全匹配的组件 → 直接用
- UI 框架有对应组件 → 用框架组件，覆盖样式
- 无匹配 → 新建组件，标注 `⚠ 新增组件`

常见错误规避 → 按需读 `references/pitfalls.md`

## 第 5 步：生成代码 + 自检

### 代码生成顺序

1. **页面容器** — 最外层 view/div，设置页面背景色、最小高度
2. **区域容器** — 按设计稿从上到下的区域划分
3. **区域内部元素** — 按元素分类逐个还原
4. **交互逻辑** — script/setup 中的状态和事件
5. **路由配置** — 如需要

### 改造现有页面

1. 读取现有文件，理解现有逻辑
2. 列出变更范围，等用户确认
3. 只修改变更项，保留现有业务逻辑

### 自检清单

**结构完整性：**
- [ ] 页面背景色与设计稿一致
- [ ] 所有元素均已实现，无遗漏
- [ ] 无 `// ...` 省略，无设计稿中不存在的元素
- [ ] 切图资源已下载到本地并正确引用

**CSS 精确度：**
- [ ] 所有 CSS 值与设计稿一致
- [ ] 颜色粒度：每个子元素颜色匹配（非区域主色）
- [ ] 布局方向与设计稿坐标一致
- [ ] 卡片 padding 逐一计算，元素尺寸来自 layer tree
- [ ] 相邻间距从坐标差值计算，重叠元素用 absolute 定位

**四项校验复核：**
- [ ] 可见性：所有文字在父容器背景下清晰可见
- [ ] 非跨页复制：借用的值已与当前设计数据交叉验证
- [ ] 富文本：含数字/状态词的文本已逐字核对颜色
- [ ] 切图尺寸：显示尺寸与 slice size 一致，含背景切图不加额外背景

## 禁止事项

| 禁止 | 原因 |
|------|------|
| 简化渐变为纯色 | 破坏还原度 |
| 省略代码用占位符 | 输出不可用 |
| 自行发明 API 路径 | 产生 bug |
| 猜测组件 props 名称 | 使用错误 API |
| 添加设计稿中不存在的元素 | 还原度失真 |
| 忽略项目现有组件直接写原生 HTML | 重复造轮子 |
| 用区域主色覆盖子元素颜色 | 颜色不精确 |
| 凭直觉判断布局方向 | 位置错误 |
| 跳过页面背景色提取 | 色调丢失 |
| **从其他页面复制 CSS 值时未与当前设计数据验证** | **值可能不同** |
| **文字颜色与父容器背景色相同时未标记告警** | **不可见问题** |
| **对含数字/状态词的文本不核对逐字颜色** | **富文本丢失** |

## 性能规范

**遵守以下约束，避免上下文爆炸和耗时过长：**

1. **中间产物不输出全文** — 元素清单、CSS 对照表用紧凑格式或直接内化到代码注释
2. **智能确认节奏** — 默认连续执行、仅在发现歧义时暂停。但在以下情况恢复中间确认点（元素分类后、代码生成前），给用户审查机会：
   - 设计稿元素 > 30 个（复杂度高，错误代价大）
   - 冲突速查表中有 ≥ 3 条冲突项
   - 设计稿类型为"改造现有页面"（改动影响已有代码，需用户确认变更范围）
3. **并行执行** — 步骤 3 和步骤 4 并行；步骤 2 中的 MCP 调用并行
4. **减少 MCP 调用次数** — 设计数据和切图尽量一次获取完整
5. **参考文件按需读取** — 只在遇到不熟悉的框架/模式时才读参考文件

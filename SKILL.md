# lanhu-to-code

> v1.2.0

根据蓝湖链接或设计稿图片，自动识别项目技术栈，生成 1:1 像素级还原的前端代码。

## 核心原则

1. **像素级还原** — 所有 CSS 值必须与设计稿完全一致，不允许"近似"或"简化"
2. **技术栈适配** — 自动检测项目技术栈，生成对应框架代码
3. **优先复用** — 优先使用项目已有的组件、工具函数、设计 token
4. **完整输出** — 不省略任何代码，不用 `// ...` 占位
5. **所见即所出** — 设计稿中存在的每个可见元素都必须还原，不存在的元素绝对不能添加

---

## 流程概览

```
[1] 输入判断 + 技术栈检测 + 意图判断（并行）
[2] 获取设计数据 + 下载切图 + 布局策略扫描 + 视觉校验
[3] 设计分析（页面属性 + 元素分类 + 布局策略执行 + CSS 值 + 四项校验）
[4] 项目上下文 + 组件映射（与步骤 3 并行）
[5] 生成代码 + 自检 + 编译验证
```

---

## 第 1 步：收集输入 + 技术栈检测（并行）

### 输入判断

| 输入类型 | 识别方式 | 处理方式 |
|---------|---------|---------|
| 蓝湖 URL | 包含 `lanhuapp.com` | 调用蓝湖 MCP 获取设计数据 |
| 设计图图片 | 图片文件路径或截图 | 用视觉分析提取设计信息 |
| 纯文本描述 | 文字描述 UI | 追问用户提供设计稿 |

### 技术栈检测

通过 Bash 工具运行 `bash scripts/detect-tech-stack.sh <项目根目录>`（脚本位于 skill 安装目录下）。失败时手动检查 `package.json` → `CLAUDE.md` → `src/` 目录结构。技术栈决定代码风格 → 按需读 `references/framework-patterns.md`。

### 意图判断（新建 vs 改造）

用户说"修改/改造"或指定文件路径 → **改造模式**。否则 → **新建模式**。

## 第 2 步：获取设计数据 + 布局策略扫描

### 蓝湖 URL 调用流程

**根据 URL 类型选择调用方式：**

| URL 类型 | 特征 | 调用方式 |
|---------|------|---------|
| detailDetach | 含 `image_id` 参数 | 先 `mode: "list"` 获取 design_names，再并行调 analyze + slices |
| stage | 含 `tid` + `pid`，无 `image_id` | 直接并行调 analyze + slices，`design_names: "all"` |

**并行调用模板（一次完成数据获取）：**

```
第 1 批（如需 list）：mcp__lanhu__lanhu_design mode: "list"
第 2 批（并行）：mcp__lanhu__lanhu_design mode: "analyze" + mode: "slices"
```

**analyze 参数：** `include: ["html", "tokens", "layers", "image"]`，`design_names: "<从 list 获取的名称>"`

### 下载切图

获取 slices 后，**筛选需要下载的切图**（排除状态栏电池/WiFi 等系统 UI 小零件），批量下载到项目静态资源目录。

### 布局策略快速扫描（< 2 分钟，数据获取后立即执行）

**拿到设计稿截图后，先完成布局策略判断再深入分析。** 扫视每个区域，标记布局类型：

| 布局类型 | 判断特征 | 后续分析策略 |
|---------|---------|------------|
| **流式布局** | 元素从上到下/从左到右顺序排列，无重叠，无 Z 轴层叠 | 只读间距/尺寸/颜色/padding，不做坐标推算 |
| **重叠布局** | 元素有视觉重叠（hero 与内容区、图标"坐"在进度条上、兄弟元素交叉） | 逐元素坐标推算，absolute 定位 |

**判断方法：** 每个区域问一个问题——有没有元素互相覆盖或交叉？没有 → 流式；有 → 重叠。

**输出：** 在思考中对每个区域标记布局类型，如「导航栏：流式」「会员卡：流式」「权益网格：流式」「Hero+内容：重叠」。这个标记直接决定步骤 3 的分析深度。

### 视觉校验（与布局策略扫描同步）

先看设计稿截图，再看 tokens/标注数据：

a. **分析截图** — 用图像分析工具逐区域扫视：背景色、渐变、阴影、圆角
b. **核对 tokens** — 截图与 tokens 冲突时，以截图为准
c. **关键检查**：
   - 每个区域的背景色/渐变是否都有对应 token？缺失的从截图提取
   - 卡片圆角、按钮圆角在截图中是否可见？tokens 无此值时从截图提取，**禁止从其他页面推断**
   - **渐变色处理**：标注只标 `gradient:linear` 无具体色值时，从视觉分析提取近似渐变，代码注释中标注来源

### 图片输入流程

用视觉分析提取布局、颜色、字体信息 → 后续步骤降级处理（参见 `references/image-input-guide.md`）。

## 第 3 步：设计分析

**步骤 3.1~3.7 全部内化到思考过程中，不产出中间文档/表格/清单。** 唯一需要输出文字的情况：冲突速查表触发条件满足时（≥3 条冲突）。

### 3.1 页面全局属性

从设计稿最底层提取页面背景色、最小高度、设计稿宽度、适配方案（rpx/px）。

### 3.2 元素分类

| 分类 | 判断依据 | 实现方式 |
|------|---------|---------|
| DOM 文字 | 纯文本，有字体/字号/颜色 | HTML + CSS |
| DOM 形状 | 简单几何（矩形/圆/线条），纯色或渐变 | HTML + CSS |
| 切图-图标 | < 100px，有明确语义 | `<image>` + 切图文件 |
| 切图-背景 | 大面积，含丰富色彩/图案 | `<image>` + 背景色兜底 |

详细规则和决策树见 `references/element-classification.md`。

### 3.3 CSS 值提取

**根据步骤 2 的布局策略标记，按区域执行对应深度的分析。**

#### 所有布局共用的规则

- 颜色按**最小可见单元**逐一提取（不取区域主色）
- 尺寸从 **layer tree / annotations** 提取（不凭感觉）
- 优先使用项目的 CSS 变量 / 设计 token，无对应 token 时用设计稿原始值

#### 流式布局区域 — 快速读值指南

| 需要的值 | 从哪读 | 换算 |
|---------|--------|------|
| 容器尺寸（宽/高） | layer_tree 括号内 `(702x294)` | 直接是 rpx |
| padding | sketch_annotations 中容器位置减内容位置 | 逻辑px × 2 |
| gap | 同行相邻元素的位置差 | 逻辑px × 2 |
| 文字颜色 | sketch_annotations 中 textLayer 的 color | 直接使用 |
| 字体大小 | sketch_annotations 中 textLayer 的 font-size | 逻辑px × 2 |
| 背景色 | sketch_annotations 中 shapeLayer 的 fill | 直接使用 |

**流式区域禁止做的事：** 不计算每个元素的绝对坐标，不推算 margin-top 链式累加。直接用 flex + gap + padding 实现即可。

#### 重叠布局区域 — 精确坐标计算

- 间距从**绝对坐标差值**计算：`间距 = next.top - prev.top - prev.height`
- 每张卡片 **padding 逐一计算**：`padding = (外框宽 - 内容宽) / 2`
- 重叠量精确计算（如 hero 负 margin）
- 详细公式和示例见 `references/css-value-extraction.md`

#### 渐变色提取规则

1. 标注中有具体渐变色值 → 直接使用
2. 标注只标 `gradient:linear` 无色值 → 从视觉分析提取近似渐变，代码注释标注 `/* 渐变来自视觉分析 */`
3. 切图/装饰图已含渐变 → 直接用切图，CSS 不重复写渐变
4. 文字渐变 → 使用 `background-clip: text` 实现

### 3.4 CSS 冲突速查（异常时输出）

正常情况下不输出对照表，直接内化到代码中。但遇到以下情况时，**必须输出精简版冲突表**供用户审查：

- tokens 数据与截图视觉不一致（颜色、圆角、间距）
- 从已有页面借用的值与当前设计数据不同
- 四项校验中发现异常（可见性问题、富文本变色、切图尺寸偏差）

无冲突时跳过此表，避免输出噪音。

### 3.5 四项强制校验（防止高频 bug）

**CSS 值提取完成后，必须逐条过以下校验。任何一项失败都需修正后再继续。** 详细规则和示例见 `references/css-value-extraction.md`。

| 校验 | 核心规则 | 常见问题 |
|------|---------|---------|
| **A. 可见性** | 文字颜色 ≈ 父容器背景色 → 标记 ⚠，检查兄弟节点是否有背景 shape | 白字白底：文字实际有渐变 pill 包裹 |
| **B. 跨页复制** | 借用已有页面的具体 CSS 值前，必查当前 tokens/annotations 是否有不同值 | 不同设计稿的圆角/间距可能不同 |
| **C. 富文本** | 含数字/状态词的文本，用图像分析逐字核对颜色，拆分为多个元素 | "3人"应为红色但标注整体灰色 |
| **D. 切图尺寸** | 下载切图后检查是否含背景，决定 CSS 尺寸和背景写法 | artboard 含背景直接展示，shapeLayer 需额外背景容器 |

### 3.6 Layer Tree 结构还原

- 父子关系 → 嵌套的 HTML 容器
- 兄弟关系 + 视觉重叠（步骤 2 标记为「重叠」的区域）→ `position: relative` + `position: absolute`
- 兄弟关系 + 顺序排列（标记为「流式」的区域）→ flex 布局
- 不能把所有元素都放进 flex column 顺序排列——重叠区域必须用 absolute

常见布局模式速查 → `references/layout-patterns.md`

## 第 4 步：项目上下文 + 组件映射（与第 3 步并行）

### 读取项目文档

| 用途 | 优先查找 | 兜底 |
|------|---------|------|
| 组件库 | `docs/components.md` | 扫描 `src/components/` |
| 编码规范 | `docs/dev-spec.md` | 读现有文件推断风格 |
| 设计 token | `src/styles/variables.scss` → `tailwind.config.*` | 用设计稿原始值 |
| 路由配置 | `src/router/` → `docs/tech-spec.md` | 生成新路由 |

**注意校验 B：参考已有页面的代码风格（组件结构、命名规范），但具体 CSS 值（圆角、间距、尺寸）必须以当前设计稿数据为准。**

### 组件映射

- 项目有完全匹配的组件 → 直接用
- UI 框架有对应组件 → 用框架组件，覆盖样式
- 无匹配 → 新建组件，标注 `⚠ 新增组件`

常见错误规避 → 按需读 `references/pitfalls.md`

## 第 5 步：生成代码 + 自检 + 编译验证

### 代码生成顺序

1. **页面容器** — 最外层 view/div，设置页面背景色、最小高度
2. **区域容器** — 按设计稿从上到下的区域划分
3. **区域内部元素** — 按元素分类逐个还原
4. **交互逻辑** — script/setup 中的状态和事件
5. **路由配置** — 如需要

### 改造现有页面

1. 读取现有文件，理解现有逻辑
2. 列出变更范围，等用户确认
3. 只修改变更项，保留现有业务逻辑

### 自检清单

**结构完整性：**
- [ ] 页面背景色与设计稿一致
- [ ] 所有元素均已实现，无遗漏
- [ ] 无 `// ...` 省略，无设计稿中不存在的元素
- [ ] 切图资源已下载到本地并正确引用

**CSS 精确度：**
- [ ] 所有 CSS 值与设计稿一致
- [ ] 颜色粒度：每个子元素颜色匹配（非区域主色）
- [ ] 布局策略已按区域分类（流式 vs 重叠）
- [ ] 流式区域：间距/padding 从标注直接读取，用 flex + gap 实现
- [ ] 重叠区域：坐标差值计算间距，padding 逐一计算，absolute 定位
- [ ] 渐变色来源已标注（标注精确值 or 视觉分析近似值）

**四项校验复核：**
- [ ] 可见性：所有文字在父容器背景下清晰可见
- [ ] 非跨页复制：借用的值已与当前设计数据交叉验证
- [ ] 富文本：含数字/状态词的文本已逐字核对颜色
- [ ] 切图尺寸：显示尺寸与 slice size 一致，含背景切图不加额外背景

### 编译验证

- uni-app：`npx uni build -p mp-weixin`（用 build 不用 dev，dev 会 hang）
- Vue/React/Vite：`npm run build`（不用 dev server，会 hang）
- 仅验证编译通过无语法错误，不做运行时验证

## 禁止事项

| 禁止 | 原因 |
|------|------|
| 简化渐变为纯色 | 破坏还原度 |
| 省略代码用占位符 | 输出不可用 |
| 自行发明 API 路径 | 产生 bug |
| 猜测组件 props 名称 | 使用错误 API |
| 添加设计稿中不存在的元素 | 还原度失真 |
| 忽略项目现有组件直接写原生 HTML | 重复造轮子 |
| 用区域主色覆盖子元素颜色 | 颜色不精确 |
| 凭直觉判断布局方向 | 位置错误 |
| 跳过页面背景色提取 | 色调丢失 |
| **从其他页面复制 CSS 值时未与当前设计数据验证** | **值可能不同** |
| **文字颜色与父容器背景色相同时未标记告警** | **不可见问题** |
| **对含数字/状态词的文本不核对逐字颜色** | **富文本丢失** |
| **对流式布局区域做逐像素坐标推算** | **浪费时间，flex 即可还原** |
| **对所有区域不区分布局类型，统一深度分析** | **流式区域过度分析，耗时翻倍** |

### 允许事项

| 允许 | 条件 |
|------|------|
| 用视觉分析近似渐变替代 | 标注完全缺失渐变色值（仅标 `gradient:linear` 无具体色值）的兜底场景，已用图像分析工具提取，代码注释标注来源 |
| 流式布局区域不计算绝对坐标 | 步骤 2 已标记为「流式」，用 flex + gap 即可 |

## 性能规范

**遵守以下约束，避免上下文爆炸和耗时过长：**

1. **中间产物不输出全文** — 元素清单、CSS 对照表、勾选清单全部内化到思考中，不产出中间文档。唯一例外：冲突速查表触发条件满足时（≥3 条冲突）
2. **智能确认节奏** — 默认连续执行、仅在发现歧义时暂停。但在以下情况恢复中间确认点（元素分类后、代码生成前），给用户审查机会：
   - 设计稿元素 > 30 个（复杂度高，错误代价大）
   - 冲突速查表中有 ≥ 3 条冲突项
   - 设计稿类型为"改造现有页面"（改动影响已有代码，需用户确认变更范围）
3. **并行调用分组** — 每批不超过 3-4 个工具调用，避免返回数据爆炸：
   - 第 1 批：技术栈检测 + 蓝湖 MCP（list/analyze/slices）
   - 第 2 批：视觉校验（图像分析） + 项目上下文探索（Agent Explore）
   - 第 3 批：下载切图 + 生成代码 + 更新路由
4. **减少 MCP 调用次数** — 设计数据和切图尽量一次获取完整，detailDetach URL 先 list 再并行调 analyze + slices
5. **参考文件按需读取** — 只在满足触发条件时才读参考文件：

| 文件 | 何时读取 | 何时跳过 |
|------|---------|---------|
| `css-value-extraction.md` | 遇到重叠布局需要坐标计算时 | 全部是流式布局 |
| `element-classification.md` | 不确定元素分类时 | 常规分类 |
| `pitfalls.md` | 自检阶段逐条过 | 生成代码前 |
| `framework-patterns.md` | 不熟悉当前技术栈时 | 熟悉的框架 |
| `image-input-guide.md` | 输入是图片非蓝湖 URL 时 | 蓝湖 URL |
| `layout-patterns.md` | 遇到不确定如何实现的布局模式时 | 常见模式 |

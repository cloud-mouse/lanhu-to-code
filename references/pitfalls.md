# 常见失败模式和规避方法

真实项目中设计稿转代码的高频失败案例，必须在生成代码前和自检时逐一核对。

---

## 失败 1：硬编码设计 token 值（最高频）

**场景：** 蓝湖/图片给出颜色 `#7c3aed`、间距 `14px`、字号 `14px`，直接写进代码。但项目有 CSS 变量 `var(--color-primary)`、`var(--spacing-sm)`、`text-sm`。

**识别信号：** 代码中出现十六进制颜色值、px 数字字面量，而非 CSS 变量或设计 token。

```css
/* 错误 */
color: #7c3aed;
margin: 14px;

/* 正确 */
color: var(--color-primary);
margin: var(--spacing-sm);
```

**正确做法：** 先查项目的 `variables.scss` / `theme.ts` / `tailwind.config.*` 中的 token 映射表；找不到则在输出摘要中标注 `⚠ 未映射 token: #7c3aed`，不自行命名。

---

## 失败 2：重写已有组件

**场景：** 蓝湖中有搜索框，直接用 `<input>` 实现，但项目的 `<SearchInput>` 有内置防抖、清除按钮和样式规范。

**识别信号：** 生成代码中出现基础 HTML 元素（input/button/select），而项目中已有封装组件。

**正确做法：** 扫描 `src/components/` 和 UI 框架 → 找到已有组件 → 使用它。

---

## 失败 3：自行发明 API 路径

**场景：** 页面需要加载列表，代码写了 `/api/order/list`，实际接口是 `/v2/orders?type=xxx`。

**识别信号：** 代码中出现 `/api/` 开头的路径但无文档支持。

**正确做法：** 在 `docs/tech-spec.md` 或现有代码中找接口 → 找不到用 `// TODO: 补充 API 路径` 标记。

---

## 失败 4：简化渐变/阴影/边框

**场景：** 设计稿有 `linear-gradient(135deg, #667eea 0%, #764ba2 100%)`，直接用 `background: #667eea` 纯色代替。

**识别信号：** CSS 对照表中设计稿值与代码值不一致。

**正确做法：** 渐变、多阴影、复合边框必须原样保留，不允许任何简化。

---

## 失败 5：省略代码

**场景：** 用 `// ... rest of component` 或 `/* 样式同上 */` 代替实际代码。

**识别信号：** 代码中出现省略注释。

**正确做法：** 写完整代码。不确定的逻辑用 `// TODO:` 标记，但不省略任何区块。

---

## 失败 6：忽略响应式/安全区域

**场景：** 移动端页面没有处理状态栏高度、底部安全区域。

**正确做法：**
- uni-app / 小程序：使用 `uni.getSystemInfoSync()` 获取状态栏高度，用 `env(safe-area-inset-bottom)` 处理底部安全区
- H5：使用 `viewport` 单位，处理 `100vh` 在移动端的问题

---

## 失败 7：图片资源路径错误

**场景：** 代码中写 `src="./image.png"` 或使用了蓝湖 CDN 临时链接。

**正确做法：**
- 蓝湖切图保存到 `src/assets/images/` 或项目约定的静态资源目录
- 引用时使用相对路径或 `@/assets/images/xxx.png`
- 不要使用蓝湖 CDN 链接（会过期）

---

## 失败 8：忽略设计稿标注的特殊状态

**场景：** 设计稿有 hover 态、选中态、禁用态，但只生成了默认态样式。

**正确做法：** 检查设计稿中是否有标注的状态变体，逐一实现。如果蓝湖 tokens 中有 `hover`、`active`、`disabled` 等状态，都要还原。

---

## 失败 9：props 名称猜错

**场景：** 用了 `<Button type="primary">` 但实际组件的 prop 是 `variant="primary"`。

**识别信号：** 使用了组件文档中未记录的 prop 名称。

**正确做法：** props 名称必须以组件文档中的实际定义为准，不能凭经验猜测。

---

## 失败 10：忽视设计稿尺寸单位

**场景：** 设计稿使用 `px` 标注，但项目是移动端需要用 `rpx` 或 `rem`。

**正确做法：**
- uni-app：`1px ≈ 2rpx`（基于 375px 设计稿）
- 小程序：同 uni-app 换算规则
- H5 移动端：根据项目根字号配置换算，或使用 `postcss-pxtorem`
- PC 端：直接使用 `px`

---

## 失败 11：凭空发明设计稿中不存在的装饰性元素

**场景：** 设计稿中卡片没有标题行，但生成代码时自行添加了"竖条指示器 + 标题文字"（如"邀请奖励"、"邀请步骤"、"我的邀请记录"）。或者自行添加了设计稿中不存在的分割线、间距装饰。

**识别信号：** 生成的代码中有设计稿中看不到的 UI 元素。

**正确做法：** 严格只还原设计稿中实际存在的元素。如果不确定某个元素是否存在，在输出中标注 `⚠ 未在设计中确认`，不要自行添加。

---

## 失败 12：遗漏设计稿中的可见元素

**场景：** 设计稿中优惠券右侧有"立即领取"按钮，但生成代码时完全遗漏了这个元素。或者里程碑节点的渐变背景标签被遗漏。

**识别信号：** 将生成的代码与设计稿逐区域比对，发现设计稿中有但代码中没有的元素。

**正确做法：** 在步骤 2.3 元素分类中，逐元素从设计稿 layers/tokens 数据中提取所有可见元素。遗漏是最严重的还原度问题。

---

## 失败 13：页面整体背景色丢失

**场景：** 设计稿页面有明显的背景色（如渐变或特定色值），但生成的代码页面背景是默认白色。

**正确做法：** 在步骤 2.1 页面全局属性中，首先提取画布最底层的背景色/渐变，确保页面容器设置了正确的 background 属性。背景图加载期间应有背景色兜底。

---

## 失败 14：用区域主色覆盖子元素颜色

**场景：** 优惠券卡片整体色调是暖黄色，左侧金额数字实际应为红色（#D80D29），但代码中把整个卡片的文字都标为了金色（#946B41）。

**识别信号：** 同一区域内多个子元素的颜色被统一为一个值。

**正确做法：** 在步骤 2 代码生成中，颜色必须按**最小可见单元**逐一提取。每个 `<text>`、`<view>` 等元素如果颜色不同，必须分别列出，不能取区域主色作为统一值。

---

## 失败 15：凭直觉判断元素布局方向

**场景：** 设计稿中里程碑节点的标签在节点上方，但代码中放在了节点下方。或者设计稿中按钮在卡片右侧，代码中放在了左侧。

**识别信号：** 生成的代码布局与设计稿位置关系不一致。

**正确做法：** 不能凭直觉或常见 UI 模式判断元素位置。必须查看设计稿的坐标数据（蓝湖 layers 中的 x/y 值），以坐标关系为准确定上下左右位置。

---

## 失败 16：内容区与头部重叠量靠猜（最高频之一）

**场景：** 设计稿中内容卡片与 hero 头部有重叠，但猜了一个 -60rpx 的负 margin，实际应该是 -290rpx，导致内容区离顶部太远。

**识别信号：** 页面上内容卡片与 hero 区域之间有明显空白间隔，而设计稿中卡片紧贴甚至重叠 hero 底部。

**正确做法：**
1. 从设计稿 layer tree 或 sketch_annotations 找到内容区起始 Y 坐标（如 `col @0,230` 表示 y=230 逻辑像素）
2. 计算：`负margin = 内容起始Y × 2（转rpx） - hero高度`
3. 例：内容 y=230 → 460rpx，hero=750rpx → margin-top: -290rpx
4. **绝对不能凭感觉写一个 -60rpx 或 -100rpx**

---

## 失败 17：嵌套组件的层级结构未按 layer tree 还原

**场景：** 设计稿中进度条的图标"坐"在进度条上（layer tree 中它们是同层兄弟），但代码把图标放在进度条上方的一个独立 flex column 中，导致图标和进度条分离、不重叠。

**识别信号：** 某些元素应该在视觉上重叠或交叉，但代码中它们是顺序排列的独立块。

**正确做法：**
1. 仔细阅读 layer tree 的嵌套结构，理解兄弟关系 vs 父子关系
2. 如果两个元素是兄弟关系且有视觉重叠（如图标在进度条上），必须用 `position: relative` + `position: absolute` 实现重叠
3. 不要自动把所有元素都放进 flex column 顺序排列——先确认设计稿中的 Z 轴层叠关系

---

## 失败 18：不同卡片的内边距被统一为一个值

**场景：** 设计稿中奖励卡内容宽 638rpx（padding 32rpx），步骤卡内容宽 654rpx（padding 24rpx），但代码给所有卡片统一写了 padding: 32rpx。

**识别信号：** 某些卡片内容看起来偏左或偏右，与设计稿不对齐。

**正确做法：**
1. 从 layer tree 提取每张卡片的外框宽度和内部内容宽度
2. 分别计算 padding = (外框宽 - 内容宽) / 2
3. 不同卡片可能有不同 padding，必须逐一计算，不能用统一值

---

## 失败 19：元素尺寸直接用感觉值，未从设计数据反推

**场景：** 进度条高度写了 8rpx，设计稿实际是 16rpx；优惠券缺口写了 16rpx，设计稿是 8rpx；步骤图标写了 48rpx，切片实际是 80rpx。

**识别信号：** 代码中的尺寸值看起来"差不多"但设计稿上明显偏大或偏小。

**正确做法：**
1. 所有尺寸必须从 sketch_annotations 或 layer tree 中提取精确值
2. **sketch_annotations 的值是逻辑像素，× 2 才是 rpx**
3. **layer tree 的值是 @2x 像素，直接等于 rpx**（在 750rpx 基准下）
4. 切图的 size 字段就是显示尺寸（rpx），不要自行缩放
5. 切片如已包含背景（如步骤图标含圆形背景），不要再加额外的背景层

---

## 失败 20：元素间距用 margin-top 链式累加，未从绝对坐标反推

**场景：** subtitle-bar 写了 margin-top: 22rpx，但设计稿中标题底部到 subtitle 顶部只有 9rpx。原因是"感觉"需要一个合理间距就随手写了。

**识别信号：** 多个相邻元素的间距与设计稿明显不符（过大或过小）。

**正确做法：**
1. 从 sketch_annotations 取两个相邻元素的绝对 Y 坐标
2. 间距 = 下一个元素 top - 上一个元素 top - 上一个元素 height
3. 例：标题 top=197rpx height=104rpx，subtitle top=310rpx → 间距 = 310-197-104 = 9rpx
4. 用绝对坐标差值计算，不要靠感觉

---

## 失败 21：白字白底 — 文字颜色与父容器背景相同导致不可见

**场景：** 蓝湖标注返回文字颜色为 `rgba(255,255,255,1)`（白色），代码直接使用。但该文字位于白色卡片（`background: #FFFFFF`）内部，白字在白底上完全不可见。

**识别信号：** 生成的代码中，某段文字的 `color` 值与其父容器（或祖先容器）的 `background-color` 相同或极度接近。

**根因：** 蓝湖标注以 textLayer 为粒度，只记录文字本身的颜色。但文字可能拥有独立背景容器（如渐变 pill、圆角 badge），这个背景容器的信息在 layer tree 中是文字的兄弟节点，容易被忽略。

**正确做法：**
1. 对每个文字元素，对比其 `color` 与父容器 `background-color`
2. 如果两者相同或极度接近（对比度不足），**立即标记为 `⚠ 可见性问题`**
3. 检查 layer tree 中该文字元素的兄弟节点 —— 是否存在背景 shape（如 `Rectangle`、`Ellipse`、带 `fill` 的 shapeLayer）
4. 如有背景 shape → 文字需要包裹在一个带背景的容器中（如渐变 pill、圆角 badge）
5. 如无背景 shape → 文字颜色可能有误，用图像分析工具从截图中重新提取

**自检方法：** 在代码中搜索所有 `color: #FFFFFF` 或 `color: white` 的文字，逐一确认其父容器背景色不是白色。

---

## 失败 22：跨页面复制样式未验证 — 从已有页面复制的值与当前设计不一致

**场景：** 项目中已有页面 `activity/index.vue` 的卡片圆角是 `border-radius: 16rpx`。生成新页面时，直接复制了这个值。但当前设计稿的卡片圆角实际是 `32rpx`。类似问题也常见于 padding、icon 尺寸、间距等值。

**识别信号：** 生成的代码中，CSS 值与已有页面完全一致，但与当前设计稿的 tokens/annotations/截图不匹配。

**根因：** 技能的"项目上下文收集"步骤鼓励参考已有代码。但参考的是**代码结构/风格**（如组件组织方式、命名规范），不是**具体数值**。具体 CSS 值（圆角、间距、尺寸）因设计稿而异，不能跨页复制。

**正确做法：**
1. 从已有页面参考的是：代码风格、组件结构、命名规范、技术栈用法
2. 具体 CSS 值（border-radius、padding、gap、icon size 等）**必须从当前设计稿数据中提取**
3. 当 tokens/annotations 中缺失某个值时：
   - **优先**：用图像分析工具从当前设计稿截图中提取
   - **禁止**：从项目其他页面的代码中推断
4. 建立验证习惯：复用已有值前，先问"当前设计稿有明确的不同值吗？"

**自检方法：** 对比生成的 CSS 中被直接复用的值，与当前设计稿的 tokens/annotations 是否一致。

---

## 失败 23：富文本颜色丢失 — 标注无法表达同一段文字内部部分变色

**场景：** 设计稿中标题"注册并首单核销人数：3人"，其中"3人"是红色高亮，其余文字是深灰色。但蓝湖标注将整个文本作为单个 textLayer 返回，颜色标记为 `rgba(50,50,51,1)`（深灰），丢失了"3人"的红色。

**识别信号：** 文本内容包含以下特征时，需高度警惕：
- 包含数字（"3人"、"80%"、"¥199"）
- 包含状态词（"已完成"、"进行中"、"未核销"）
- 标题中包含冒号分隔的关键信息（"人数：X"）

**根因：** 蓝湖/Figma 标注以 textLayer 为粒度输出，一个 textLayer 只有一个颜色值。但实际设计中经常出现同一行文字内部部分字符颜色不同（富文本），这些信息在数据层面丢失了。

**正确做法：**
1. **触发条件**：当文本内容包含数字、状态词、或冒号后的关键信息时
2. **验证方法**：用图像分析工具对设计稿截图中该文字区域进行逐字颜色分析
3. **实现方式**：将文本拆分为多个 `<text>` 元素，分别设置颜色
4. **典型模式**：
   ```
   <!-- 错误：整体一个颜色 -->
   <text class="title">注册并首单核销人数：3人</text>

   <!-- 正确：拆分，关键数字独立标色 -->
   <view class="title">
     <text>注册并首单核销人数：</text>
     <text class="highlight">3人</text>
   </view>
   ```

**自检方法：** 扫描所有文本内容，含数字的标题/标签一律用图像分析工具核对逐字颜色。

---

## 失败 24：切图尺寸不匹配 — 切图是否包含背景未检查导致 CSS 尺寸错误

**场景：** 设计稿中步骤图标是 80x80 的圆形（含 #FFF4E6 圆形背景 + 内部图形），下载的切片是完整的 80x80 图片。但代码中参考了已有页面的写法（`icon: 40rpx` + `wrap: 80rpx` + `background: #FFF4E6`），导致图标缩小到一半，还多了一层背景色。

**识别信号：** 切图在页面上显示过小，或出现双重背景（切图自带的 + CSS 添加的）。

**根因：** 不同设计稿的切图内容不同 — 有的切图只含图形（需 CSS 加背景），有的切图含图形+背景（直接全尺寸展示）。没有在下载切图后检查其内容，盲目参考了已有页面的写法。

**正确做法：**
1. **下载切图后立即分析**：检查 slices 数据中切图的类型
   - `artboard` 类型：通常是完整组件切片，**含背景**
   - `shapeLayer` 类型：通常是纯图形切片，**不含背景**
2. **根据分析结果决定 CSS 写法**：

   ```
   情况 A：切图含背景（如 80x80 artboard）
   → CSS: width: 80rpx; height: 80rpx; （不加额外背景）

   情况 B：切图不含背景（如 40x40 shapeLayer）
   → CSS: 外层容器 80rpx + background: #FFF4E6; 内层图标 40rpx
   ```

3. **禁止参考其他页面的 icon 尺寸写法** — 每个设计稿的切图内容可能不同，必须以当前切图的实际内容为准

**自检方法：** 对每个切图元素，检查 CSS 中的尺寸是否与 slice 的 size 字段一致。含背景的切图不应有额外的 CSS 背景色。
